# Bug audit — daemon/session lifecycle "one-off" bugs (2026-07-11)

Triggered by two user reports on 0.4.0: `/resume` freezes the TUI, and new
`agenc` invocations appear pinned to the folder of the first session. Both
were root-caused; the sweep then found a family of related defects sharing
one structural cause.

**Structural root cause:** the daemon hosts N sessions in one process, but a
large set of "moved-source" utilities still assume one session per process —
module-level singletons, `process.env` snapshots frozen at daemon start, and
`process.cwd()` fallbacks that resolve to wherever the daemon happened to be
launched. `bootstrap/state.ts` (`STATE` singleton) is the hub; ~36 files
import it.

Severity key: **P0** user-visible breakage now · **P1** breaks in common
multi-session/multi-shell use · **P2** correctness hazard.

**STATUS (2026-07-11, late):** findings 1, 2, 4–11 are FIXED on main (see the
per-finding status lines); finding 3's residue is cleaned up (41,409 → 7
project dirs, 17 GB → 247 MB) and current suites measured clean; finding 12
(added late) is open.

---

## 1. `/resume` freezes the TUI — P0

**FIXED:** bounded streaming preview (`readFirstUserPreview` rewritten, no
full-file reads), `readThread({includeHistory})` removed from the picker
loop, legacy-registry import memoized per store instance, index-only
registry reads skip the exclusive directory lock, picker scan cap lowered
to 2,000.

**Symptom:** typing `/resume` locks the UI (no repaint, no keystrokes) for
seconds to minutes.

**Root cause:** the picker computes its entire session list **synchronously
on the Ink main thread before rendering anything** —
`runtime/src/commands/resume.ts:263` (`listResumableSessions(ctx.cwd)`) runs
before `openResumeMenu` at `:265`, with no await/yield anywhere inside.
Compounding stalls, ranked:

1. **N+1 full-transcript reads** — `resume.ts:156` calls
   `store.readThread({includeHistory: true})` per candidate: the entire
   rollout JSONL (MB–tens of MB) is `readFileSync`'d and parsed line-by-line
   just to derive a one-line preview; `readFirstUserPreview` (`resume.ts:67`)
   can read the same file again.
2. **O(N²) legacy-registry re-import** — every `readThread` →
   `readRegistry(includeLegacy=true)` re-runs `importLegacyRegistry`
   (`store.ts:1240`): full `sessions/` + `archived_sessions/` walks with
   per-file `statSync`/head reads **and SQLite writes**, repeated per listed
   thread.
3. **Synchronous 30 s lock busy-wait** — `withRegistryLock`
   (`store.ts:1130-1169`) spins `Atomics.wait`-style (`sleepSync(25)`) up to
   30 s while the daemon holds the registry lock (which its own comment says
   can last seconds during rollout writes). This is the hard multi-second
   freeze independent of store size.
4. **Unbounded scan** — empty-preview sessions are skipped but the loop keeps
   paging toward `MAX_SCAN_FILES = 10_000` (`resume.ts:29,169`), a full
   transcript read each, to fill a 20-row picker.
5. Same defects in the CLI `-c`/`--resume` path
   (`bin/resume-session.ts:42-45`, limit 10 000; `findConvIdGlobally` at
   `:115` additionally walks all of `~/.agenc/projects`).

**Measured amplifier on this machine:** `~/.agenc/projects` held **41,409
project dirs / ~44,000 rollout JSONLs / 17 GB** (see finding 3).

**Fix set (lowest risk first):** `includeHistory: false` + the existing
streaming `readFirstUserMessageFromRollout` (`store.ts:1661`) for previews;
render the picker first and stream rows in; lock-free/read-only registry
reads for the picker + memoize `importLegacyRegistry` per store instance;
lower the picker scan cap; apply the same to `resume-session.ts`.

## 2. Sessions pinned to the first workspace — P0 (latent on 0.4.0)

**FIXED:** precedence flipped at `bootstrap.ts:844` (`options.cwd` first);
the bootstrap contract test now asserts cwd beats a stale AGENC_WORKSPACE.

**Symptom:** running `agenc` from folder B operates in folder A (first
session's folder).

**Root cause:** inverted precedence at **`runtime/src/bin/bootstrap.ts:844-845`**:

```ts
const workspaceRoot =
  resolveWorkspaceFromEnv(env) ?? options.cwd ?? process.cwd();
```

`env` is the daemon's environment **frozen at daemon start**. A stale
`AGENC_WORKSPACE` in it beats the correct per-session `options.cwd` the
client always sends (verified: client → `agent.create` cwd propagation is
correct end-to-end, `agenc.ts:3447` → `app-server-client/index.ts:83-86` →
`agent-lifecycle.ts:349` → `background-agent-runner.ts:657-670`). The
trailing `process.cwd()` is a second leak (daemon's own cwd) if cwd is ever
omitted; `agent-lifecycle.ts:319` `#defaultCwd = () => process.cwd()` is the
same hazard. The correct ordering already exists at
`permissions/trust/project-trust.ts:515-517` (`options.cwd` first).

**Repro status:** on 0.4.0 the `--no-tui` path resolved correctly in live
testing (session in the invoking folder; trust prompt also used the correct
cwd). The user's symptom most plausibly came from (a) the long-lived daemon
running 0.2.0-era code until 2026-07-11 (the `current` symlink pointed at
0.2.0), and/or (b) `AGENC_WORKSPACE` frozen into a previous daemon env. The
inverted line is present in 0.4.0 and will re-trigger whenever
`AGENC_WORKSPACE` is exported in the daemon's launch shell.

**Fix:** flip to `options.cwd ?? resolveWorkspaceFromEnv(env) ??
process.cwd()`; optionally make `agent.create` require an explicit cwd so a
missing client cwd fails loudly instead of inheriting daemon state.

## 3. Test harnesses pollute the real `~/.agenc` — P0 (disk/perf)

**CLEANED UP:** 41,409 → 7 dirs (the 5 real projects + 2 recent), 17 GB →
247 MB. Both vitest (`vitest.setup.ts`) and the bun runner already force a
hermetic AGENC_HOME. Leak measured post-cleanup: a full vitest run + bun
suite + build created ZERO new project dirs (still 7), so current suites are
clean — the residue was historical (pre-hermeticity test eras and external
tmux-driven QA loops that drove the real installed binary against the real
home). Remaining rule: anything driving the REAL agenc binary (QA loops,
manual e2e) must set AGENC_HOME to a temp dir.

**Evidence:** 41,409 dirs under `~/.agenc/projects`, of which **39,276**
match TUI e2e/test slugs (`-tmp-user-1000-agenc-tui-e2e-slim-*` etc.),
totaling ~17 GB. The e2e/visual-QA loops created real sessions against the
user's `AGENC_HOME` instead of an isolated one.

**Impact:** direct amplifier of finding 1; disk bloat; `findConvIdGlobally`
and any projects-wide walk degrade linearly.

**Fix:** e2e harnesses must set `AGENC_HOME`/`AGENC_CONFIG_DIR` to a temp dir
(and assert it); add a guard that refuses to run test scenarios against the
default home. Cleanup: prune `-tmp-*`/e2e slugs from `~/.agenc/projects`
(operator action; not performed automatically by this audit).

## 4. Env overrides don't reach daemon sessions (stale keys/PATH/proxy) — P1

**FIXED:** `collectDaemonClientEnvOverrides` allowlist (provider keys, base
URLs, `AGENC_MODEL/PROVIDER/PROFILE`, proxy vars, PATH; `AGENC_WORKSPACE`
deliberately excluded) now forwarded by the prompt, one-shot, and
`agent start` paths; set-only semantics keep daemon fallbacks.

`app-server-client/index.ts:64` forwards **only `AGENC_MCP_SERVERS`** with
`agent.create`; `agenc agent start` and the one-shot path forward nothing.
Every session therefore resolves providers/keys/proxies/PATH from the env
frozen at daemon start (`background-agent-runner.ts:653-662` merge is dead
code for everything else; env beats config.toml per `config/env.ts`).
Concrete: rotate `XAI_API_KEY` in a new shell → sessions keep using the old
one until the daemon is killed. Bash-tool children inherit daemon
`process.env` (`utils/subprocessEnv.ts:123`) so venv/nvm activation in the
new shell is invisible. **Fix:** allowlist of client-side env overrides
(provider keys, `AGENC_*`, `PATH`, proxy vars) collected per `agent.create`.

## 5. Lifecycle hooks are process-global — sessions fire each other's hooks — P1

**FIXED:** per-session `LifecycleHookRegistry` on each session's
hooksService; dispatchers resolve the session registry (explicit or ambient
ALS) plus true process-global hooks. Regression: two sessions' SessionEnd
hooks no longer cross-fire.

`llm/hooks/registry.ts:115` module singleton; per-session configured hooks
(SessionStart/SessionEnd/PreCompact/…, closures over that session's
cwd/shell, `bootstrap-services.ts:385-402,683-702`) all land in it; dispatch
reads the global (`llm/hooks/dispatcher.ts:231`). With sessions A and B
concurrent, B's SessionEnd runs A's `cleanup.sh` with A's cwd against B's
session id; both sets double-fire on compacts. **Fix:** move lifecycle hooks
into per-session `hooksService` (Stop/PreToolUse already are) or pass hooks
explicitly to `dispatch*`.

## 6. Agent-role registry global with once-per-cwd load guard — P1

**FIXED:** markdown roles namespaced by cwd with mtime-signature
invalidation (role edits apply to new sessions without daemon restart);
built-ins stay global; cwd-less call sites keep legacy fallback.

`agents/role.ts:344-350` one registry Map per daemon; `registerAgentRole`
overwrites by name; `loadMarkdownAgentRoles` never reloads a cwd
(`loadedMarkdownRoleCwds`), default cwd = daemon `process.cwd()`
(`role-definitions.ts:31`). Two projects defining `.agenc/agents/reviewer.md`
→ last bootstrap wins for both; editing a role .md requires a daemon restart
to take effect. **Fix:** per-session registries or cwd-namespaced entries
with mtime invalidation.

## 7. Invoked-skills tracking shared across sessions — P2

**FIXED:** invoked-skill records keyed by session (conversation id stamped
by the Skill tool); both the local-loader store and the bootstrap/state
parallel store.

`skills/local-loader.ts:167,189,1090-1103` — module-level map keyed by the
constant `"__main__"` for every daemon session (recorded at
`bin/model-facing-tools.ts:1444`, reported into each session's skill snapshot
via `skillsForConfig` at `:1222`). Session B sees session A's invoked skills
as already loaded. Same issue in the parallel store at
`bootstrap/state.ts:1377`. **Fix:** key by `${sessionId}:${agentId}`.

## 8. `/compact` (and rewind) hit a flat 30 s RPC timeout that doesn't cancel — P1

**FIXED:** compact/rewind added to `LONG_RUNNING_DAEMON_METHODS`; the
client timeout path now emits `request.cancel` before dropping the pending
request.

Only `message.stream` is in `LONG_RUNNING_DAEMON_METHODS`
(`app-server/agent-cli.ts:139-151`); `session.partialCompactFromMessage` is
an LLM summarization that easily exceeds 30 s on large transcripts/slow
models. The client timeout rejects **without** `request.cancel`
(`agent-cli.ts:952-958`), so the daemon keeps compacting; a retry gets
"a turn is currently in flight" and a surprise `context_compacted` lands
later. **Fix:** add compact/rewind to the long-running set; emit
`request.cancel` on timeout.

## 9. Gemini access token sticky in daemon env — P2

**FIXED:** hydration marker (`AGENC_GEMINI_TOKEN_HYDRATED`, githubModels
pattern): hydrated values refresh from secure storage per request;
user-provided env values are never overwritten.

`utils/geminiCredentials.ts:32-41` hydration early-returns when
`process.env.GEMINI_ACCESS_TOKEN` is set, and it runs inside the daemon
(`services/api/openaiShim.ts:2500`): an expired token blocks its own refresh
for **all** sessions until daemon restart. **Fix:** resolve per-request from
secure storage; never persist into `process.env`; invalidate on 401.

## 10. Cross-provider model poisoning of the process-global activeConfigModel — P1

**FIXED** (root cause revised after deeper investigation). Live repro on
0.4.0: `AGENC_PROVIDER=grok agenc --no-tui "…"` sent
**`qwen3-coder-next-fp8` to api.x.ai** → 404. Static resolution
(`resolveStartupSelection`, `configuredModelForProvider`) is provably
correct — probes with the real config resolve `grok/grok-4.5`. The true
mechanism is runtime state: a **provider-only switch** (e.g. `/grok-login`'s
auto `applyProviderSwitch`) is staged for the NEXT turn, so
`setAgentModel` → `#resolveEffectiveConfigModel`
(`background-agent-runner.ts:1557`) read the PRE-switch session selection
and backfilled its model, writing the mixed pair
`{provider: "grok", model: "qwen3-coder-next-fp8"}` into the process-global
`activeConfigModel` — which later daemon sessions inherit (another instance
of the finding-11 family). Fix: the live-session model is only used as
backfill when its provider matches the effective provider; regression test
in `background-agent-runner.config-model-state.test.ts`. (The env-forwarding
fix in finding 4 additionally makes `AGENC_MODEL`/`AGENC_PROVIDER` reach the
daemon.)

## 11. `currentRuntimeSession` global fallback — P2 hazard

**FIXED:** live-session tracking; with >1 bootstrapped session and no ALS
scope, access throws a descriptive error instead of silently binding to the
last session; single-session processes keep the fallback.

`session/current-session.ts:5,11-12` module-level fallback overwritten by
every bootstrap (`bin/bootstrap.ts:1406`); ALS scoping only wraps `runTurn`.
Consumers that run outside a turn's ALS context (`utils/forkedAgent.ts:532`,
`utils/hooks/execAgentHook.ts:178`, `tools/AgentTool/runAgent.ts:376`,
`tasks/LocalMainSessionTask.ts:356`) bind to whichever session bootstrapped
last. No guaranteed trigger traced; fix is to remove the global fallback in
the daemon and fail loudly.

## 12. Unbounded daemon log rotation — P2 (open)

Found during investigation: `~/.agenc/daemon.log.1` is **1.7 GB** (rotated
once, then grew unbounded; full of repeated EPIPE stacks). The daemon log
sink rotates to a single `.1` with no size cap or retention. Fix: size-capped
rotation with a small retention count, and rate-limit repeated identical
write-error stacks.

## 13. "Working…" spinner never stops after a terminal provider error — P0 (FIXED)

**Symptom** (user-reported on 0.4.1): provider unreachable (local vLLM down →
`connection_refused`); the error renders in the transcript but the TUI cycles
"✢ Working…" forever. Live repro: single error event, spinner still on 135 s
later, daemon shows NO active agent (its side ended correctly).

**Root cause:** an error-terminated turn never reaches the TUI as
`turn_complete` — run-turn's `turn_complete(stopReason:"error")` is remapped
by the daemon (`background-agent-runner.ts:3941` → `run_error` →
`agent_status:error`) and arrives as a transcript `error` event. The
transcript reducer (`tui/session-transcript.ts` `case "error"`) pushed the
message but never cleared `isStreaming`; `App.tsx` `isLoading` ORs
`transcript.isStreaming` with the (already-hardened) active-turn flag, so the
latched reducer flag won. An earlier fix (`825ec4808`) hardened
`noteDaemonActivity` for exactly this event but missed the reducer — the
half-applied-pattern trap.

**FIXED:** `case "error"`/`case "stream_error"` are now turn-terminating in
the reducer (clear streaming state, preserve partial text like
`turn_aborted`). Regression test:
`tests/tui/session-transcript.error-ends-turn.test.ts` (revert-verified).
Verified live against a real dead-provider TUI session.

Defense-in-depth NOT taken (deliberate): bounding `reconnectWithBackoff` for
connection_refused would fail fast on a dead provider but would also break
the intended patience for a local vLLM that is restarting/reloading a model
(minutes); the 10-minute give-up budget stays.
